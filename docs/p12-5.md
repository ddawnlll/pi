# Part 1 — Phase Plan

## 0. TL;DR / Compact Mental Model

**Phase:** `P12.5`  
**One-line goal:** Make Pi capable of safely accepting, validating, queuing, resuming, and executing multi-plan zipped bundles without starting the next plan until the current plan and integration queue are clean.

**Done when:** Pi can validate a zip containing multiple phase plans, enqueue them as plan-level queue entries, run them sequentially with approval gates, pause/resume/cancel/requeue safely, survive restart, and block the next plan while integration queue is dirty.

## 2. Purpose

P12.5 creates the execution foundation needed before Pi can safely run the upcoming second-brain roadmap phases. The current system can run a plan, manage workspace execution, and use an integration queue for workspace merge/validation. However, it does not yet provide a first-class plan-level queue that can hold P13, P14, P15, P16, P17, and P18 as separate queued plan executions.

This phase adds a plan backlog runner above the existing workspace integration queue. The plan queue owns the sequence of plan executions. The integration queue continues to own workspace merge/validation. The core invariant is simple: a new plan may start only when the previous plan is terminal-successful and the integration queue is clean.

P12.5 also adds zipped plan bundle intake. A user should be able to upload a zip such as `second-brain-plans.zip`, have Pi safely extract and validate all included phase plans, generate queue entries, require approval, and then run the plans one by one.

This phase intentionally avoids building the full P13 orchestrator daemon. P12.5 provides the safe queue substrate that P13 will later automate.

## 4. Background / What Was Wrong

The system can run plans, but large roadmap execution still requires too much manual work. If P13–P18 are produced as separate phase plans, the user must currently upload, validate, approve, and run them manually one by one. That makes overnight execution difficult.

There is also a conceptual mismatch between two queue layers:

```text
PlanQueue
  owns the sequence of plan executions

IntegrationQueue
  owns workspace commit merge and validation inside one plan execution
````

The integration queue already provides strong workspace-level safety, but there is no plan-level queue with persistent backlog state, bundle-level validation, idempotent resume, duplicate detection, approval gating, and dashboard visibility.

Without P12.5, running zipped multi-plan bundles would be unsafe because the system could accidentally start a later phase while a previous phase still has blocked integration work, dirty queue state, merge conflicts, or stale approval.

## 7. Workstreams

### P12.5.A — Queue Domain Model Cleanup

**Goal:** Define clear queue concepts and prevent PlanQueue / IntegrationQueue ambiguity.

**Requirements:**

* Add explicit domain types for `PlanQueue`, `PlanQueueEntry`, `PlanBundle`, `PlanBundleManifest`, `PlanQueueStatus`, and `PlanQueueAuditEvent`.
* Document the two queue layers:

  * PlanQueue = sequence of plan executions.
  * IntegrationQueue = workspace merge/validation queue inside a plan.
* Define terminal, non-terminal, blocked, dirty, and clean states.
* Add queue invariants as code comments and tests.

**Acceptance Criteria:**

* Plan queue types exist and compile.
* Type names do not collide with existing IntegrationQueue types.
* Dirty integration states are clearly represented.
* Tests verify clean vs dirty state classification.
* Documentation explains the two queue layers.

**Isolation & Parallelism Notes:**

* Can run first.
* Mostly type/docs/test work.
* May run with P12.5.K only if files do not overlap, but safer as foundation.

### P12.5.B — Plan-Level Queue Store

**Goal:** Persist plan-level backlog state crash-safely.

**Requirements:**

* Add JSON-backed plan queue store under `.pi/plan-queue.json`.
* Store entries with:

  * entry id
  * phase
  * title
  * plan file path
  * plan content hash
  * bundle id if applicable
  * status
  * priority
  * dependencies
  * active plan execution id if running
  * validation summary
  * createdAt / updatedAt / startedAt / completedAt
  * error / blocked reason
* Support append, update, load, list, find duplicate, and atomic save.
* Do not require PostgreSQL.
* Keep future interface compatible with PostgreSQL-backed implementation.

**Acceptance Criteria:**

* Queue state persists across process restart.
* Duplicate hash + phase cannot be enqueued twice unless explicitly requeued.
* Atomic writes prevent truncated queue files.
* Corrupt queue file produces safe error and does not start plans.
* Unit tests cover append/update/load/duplicate behavior.

**Isolation & Parallelism Notes:**

* Depends on P12.5.A.
* Can run with API/dashboard work after interfaces stabilize.
* No integration queue mutations.

### P12.5.C — Plan Queue Runner

**Goal:** Run queued plans sequentially and safely.

**Requirements:**

* Add `PlanQueueRunner` service.
* Runner selects the next ready plan entry by:

  * dependency completion
  * queue priority
  * FIFO within priority band
  * approval state
  * integration queue cleanliness
* Start at most one plan execution per project.
* Reuse existing `runPlan()` path instead of bypassing parser/doctor.
* Persist active execution id on the queue entry.
* Poll or subscribe until the active execution reaches terminal status.
* Mark entry complete/failed/blocked/cancelled based on execution result.
* Do not start next entry if integration queue is dirty.

**Acceptance Criteria:**

* Runner starts first ready entry.
* Runner never starts two plans for same project.
* Runner blocks when previous plan is non-terminal.
* Runner blocks when integration queue is dirty.
* Runner advances to next entry after successful plan and clean integration queue.
* Tests cover sequential execution and dirty gate behavior.

**Isolation & Parallelism Notes:**

* Depends on P12.5.B and P12.5.D.
* Must be serialized per project.
* Does not increase workspace parallelism.

### P12.5.D — Integration Queue Cleanliness Gate

**Goal:** Provide a reusable gate that determines whether it is safe to start the next plan.

**Requirements:**

* Add `isIntegrationQueueClean(projectRoot)` helper/service.
* Treat these states as dirty:

  * `queued`
  * `merging`
  * `validating`
  * `blocked`
  * `conflict`
  * `failed` unless explicitly dismissed
  * `isProcessing: true`
  * `paused: true` with non-terminal entries
* Treat all entries `merged` or no entries as clean.
* Surface human-readable blocker reason.
* Integrate gate into PlanQueueRunner and bundle enqueue preview.

**Acceptance Criteria:**

* Gate blocks next plan for queued workspace integration.
* Gate blocks next plan for conflict.
* Gate blocks next plan for validation blocked.
* Gate allows next plan only when queue is empty or all entries are safely terminal-clean.
* Tests cover every queue status.

**Isolation & Parallelism Notes:**

* Depends on P12.5.A.
* Required by P12.5.C.
* Can run in parallel with P12.5.B after domain model stabilizes.

### P12.5.G — Crash Recovery and Idempotent Resume

**Goal:** Ensure plan queue execution survives process restart.

**Requirements:**

* On startup, load `.pi/plan-queue.json`.
* Detect entries stuck in `running`.
* Reconcile `activePlanExecId` with active execution registry / persisted execution state.
* If execution is still recoverable, adopt/resume.
* If execution is missing, mark entry `blocked` with recovery reason.
* Ensure retry/requeue creates a new attempt without duplicating original entry identity.
* Add queue audit events for recovery decisions.

**Acceptance Criteria:**

* Restart during queued state preserves queue.
* Restart during running state does not double-start the plan.
* Missing active execution marks entry blocked, not complete.
* Requeue after failure creates clear attempt history.
* Tests simulate restart scenarios.

**Isolation & Parallelism Notes:**

* Depends on P12.5.B and P12.5.C.
* Should not run concurrently with queue runner implementation on same files.

## 10. Rollback Playbook

**Trigger conditions:**

* Plan queue starts a second plan while another is running.
* Plan queue starts next plan while integration queue is dirty.
* Bundle extraction writes outside scoped temp directory.
* Duplicate plan execution occurs after crash recovery.
* Queue controls bypass executor-mediated state mutation.
* Dashboard shows incorrect queue status.
* Queue state file corruption causes unsafe execution.
* Zip validation accepts unsafe files.
* npm validation fails in queue-related packages.

**Rollback procedure:**

1. Disable PlanQueueRunner auto-run.
2. Disable zipped bundle enqueue endpoint.
3. Keep bundle validation endpoint read-only if safe.
4. Fall back to single-plan manual upload/run.
5. Preserve `.pi/plan-queue.json`, `.pi/bundles/`, and execution artifacts for debugging.
6. Mark active queue entry blocked if recovery state is uncertain.
7. Do not delete bundle temp directories with raw destructive commands.
8. Revert P12.5 commits independently if needed.
9. Keep existing IntegrationQueue behavior unchanged.
